1. 要复刻的不是“画几行字”，而是一条完整流水线

Bilibili Web 弹幕引擎可以拆成 8 个模块。真正影响流畅度的是前 5 个模块，渲染只是最后一步。

模块 职责 复刻重点
View 元信息 取分段配置、弹幕设置、屏蔽规则、智能过滤配置、特殊弹幕资源。 DmWebViewReply 必须解析完整，不能只拿 segment 数。
分段弹幕 按播放进度请求 protobuf 弹幕片段。 首屏分段特殊处理，避免一次拉 360 秒造成解析和布局峰值。
过滤器 按 AI 权重、类型、颜色、特殊弹幕、屏蔽词、举报过滤来裁剪。 默认开启智能过滤，不应默认全量显示。
密度控制 控制同屏数量、区域高度、低权重弹幕。 dmarea、dmdensity 会反向影响 AI 阈值。
轨道分配 滚动、顶部、底部分别排轨，避免碰撞。 滚动轨道检查尾部碰撞；固定轨道按停留时间复用。
渲染层 DOM/CSS 或 native canvas 绘制文字、描边、透明度、颜色。 字体、描边、透明度和缓存策略会明显影响性能。
遮罩层 智能防挡/蒙版遮挡弹幕。 独立于普通弹幕，但会增加合成成本。
监控埋点 记录请求、加载耗时、失败、弹幕量。 复刻时要内置性能指标，否则卡顿无法定位。

3. 数据请求策略

3.1 View 请求

官方 Web 端先请求 view 元信息，URL 形态如下：

GET //api.bilibili.com/x/v2/dm/web/view
  ?type=1
  &oid={cid}
  &pid={aid}
  &duration={videoDurationSeconds}
  &without_subtitle=true

如果存在贴片/广告片段，官方会带 context_ext，并对 command 弹幕和 post panel 的时间做偏移修正。普通 UGC 可以先不实现这部分。

3.2 Segment 请求

普通分段请求走 wbi 路径：

GET //api.bilibili.com/x/v2/dm/wbi/web/seg.so
  ?type=1
  &oid={cid}
  &pid={aid}
  &segment_index={index}

关键细节：官方 Web 首两次请求不是简单按完整 360 秒段取，而是对第一段做拆分：

// 复刻等价逻辑
function buildSegmentUrl(index, aid, cid, preOffsetMs = 0) {
  const base = `//api.bilibili.com/x/v2/dm/wbi/web/seg.so?type=1&oid=${cid}&pid=${aid}`;

  if (index < 2) {
    const ps = index === 1 ? 0 + preOffsetMs : 120_000 + preOffsetMs;
    const pe = index === 1 ? 120_000 + preOffsetMs : 360_000 + preOffsetMs;
    return `${base}&segment_index=1&pull_mode=1&ps=${ps}&pe=${pe}`;
  }

  return `${base}&segment_index=${index}`;
}

这条策略很重要。它说明 B 站首屏加载优先拿 0~120s，而不是一开始就解析完整 360 秒段。复刻时建议保留这个策略：

  • 进入视频：先请求 view。
  • 首屏：请求 segment_index=1&pull_mode=1&ps=0&pe=120000。
  • 播放进入 90 秒后：预取 120000~360000。
  • 后续：按标准 segment index 请求。

4. Protobuf 字段

官方 bundle 内置了 protobuf schema。复刻时应解析这些字段，而不是只解析 progress/content/mode。

4.1 DmWebViewReply

5. 默认设置与密度策略

官方默认弹幕设置的核心值如下：

{
  dmSwitch: true,
  aiSwitch: true,
  aiLevel: 3,
  preventshade: false,
  dmask: true,
  typeScroll: true,
  typeTopBottom: true,
  typeColor: true,
  typeSpecial: true,
  opacity: 0.8,
  dmarea: 50,
  speedplus: 1,
  fontsize: 1,
  fontfamily: "SimHei, 'Microsoft JhengHei'",
  bold: true,
  fontborder: 0,
  dmdensity: 1
}

其中 aiLevel 不是固定只看服务端值。官方还根据区域和密度动态计算：

function resolveAiLevel(setting) {
  if (setting.dmarea < 100 && setting.dmarea > 0) return 3;
  if (setting.dmdensity === "most" || setting.dmdensity === "more") return 2;
  if (setting.dmdensity === "normal") return 3;
  return undefined;
}

这说明 B 站默认体验并不是全量满屏，而是：

  • 默认只显示 50% 区域。
  • 默认透明度 0.8。
  • 默认开启 AI 过滤。
  • 密度越高，AI 阈值可能越宽；区域受限时保持较强过滤。

6. 过滤器复刻

过滤顺序建议按官方行为复刻。过滤器必须在进入轨道前执行，否则会浪费测量、布局和缓存成本。

6.1 AI 权重过滤

官方静态逻辑可还原为：

function aiJudge(dm, aiLevel) {
  // 官方使用 Math.abs(weight) < level 判定为低质量/可过滤
  return Math.abs(dm.weight) < aiLevel;
}

MyBLBL 当前按 maxPositiveScore 和用户等级做阈值，这和官方不完全一致。若要贴近 B 站，建议优先采用 abs(weight) < aiLevel 的基础策略，再叠加本地用户设置。

6.2 类型过滤

复刻版过滤器：

function shouldBlockDanmaku(dm, setting, blockList, reportFilters) {
  if (dm.border) return false;

  if (matchUserBlock(dm, blockList)) return true;
  if (matchKeywordBlock(dm, blockList)) return true;
  if (matchRegexpBlock(dm, blockList)) return true;

  if (setting.aiSwitch && aiJudge(dm, setting.aiLevel)) return true;
  if (matchReportFilter(dm.text, reportFilters)) return true;

  if (setting.seniorMode && dm.weight <= 10) return true;
  if (!setting.typeScroll && isScroll(dm.mode)) return true;
  if (!setting.typeTopBottom && isTopOrBottom(dm.mode)) return true;
  if (!setting.typeSpecial && (isSpecial(dm.mode) || dm.pool === 2)) return true;
  if (!setting.typeColor && isNonWhiteColor(dm)) return true;
  if (setting.preventshade && isPreventShadeMode(dm.mode)) return true;

  return false;
}

6.3 颜色过滤

官方不是简单判断 color !== 0xffffff，还会把彩色、高级彩色、图片前后缀、颜文字等归入颜色/特殊过滤。普通复刻可以先做：

function isNonWhiteColor(dm) {
  if (dm.colorful) return true;
  if (dm.modeInfo?.font_color && dm.modeInfo.font_color.toUpperCase() !== "#FFFFFF") return true;
  return dm.color !== 0xffffff && dm.color !== 16777215;
}

8. 加载与窗口策略

推荐实现一个三层窗口：网络分段窗口、内存时间线窗口、渲染活动窗口。

class DanmakuRepository {
  private val loadedSegments = mutableMapOf<Double, List<DanmakuModel>>()

  suspend fun ensureAround(positionMs: Long) {
    val index = resolveSegmentIndex(positionMs)
    requestIfAbsent(index)
    requestIfAbsent(index + 1) // 预取
  }

  fun queryWindow(positionMs: Long, behindMs: Long, aheadMs: Long): List<DanmakuModel> {
    val start = maxOf(0, positionMs - behindMs)
    val end = positionMs + aheadMs
    return allLoadedSorted.binaryRange(start, end)
  }
}

推荐参数：

参数建议值原因

9. 轨道分配

9.1 滚动轨道

滚动弹幕的核心公式：

function rollingX(screenWidth, itemWidth, nowMs, startMs, durationMs) {
  const elapsed = nowMs - startMs;
  return screenWidth - (elapsed / durationMs) * (screenWidth + itemWidth);
}

轨道分配只需要检查每行尾部弹幕，不应扫描整行全部弹幕：

function findRollingRow(item, rows, nowMs, config) {
  const nextStart = Math.max(nowMs, item.timeMs);
  const nextWidth = item.width;

  for (const row of rows) {
    row.dropExpired(nowMs, config.durationMs);
    if (item.height > row.height) continue;

    const tail = row.tail;
    if (!tail || !willCollide(tail, nextStart, nextWidth, config)) {
      return row;
    }
  }

  return createRowIfRoom(item.height, rows, config);
}

碰撞判断建议检查“当前时刻”和“结束时刻”。如果后一条会追上前一条，不能放同一轨。

function willCollide(prev, nextStart, nextWidth, cfg) {
  const dt = nextStart - prev.startMs;
  if (dt <= 0) return true;
  if (dt >= cfg.durationMs) return false;

  const screen = cfg.screenWidth;
  const prevDistance = screen + prev.motionWidth;
  const nextDistance = screen + nextWidth;
  const tolerance = Math.min(prev.width, nextWidth) * cfg.overlapFraction;

  const prevRightNow = screen
    - prevDistance * ((cfg.nowMs - prev.startMs) / cfg.durationMs)
    + prev.width;
  const nextLeftNow = screen
    - nextDistance * ((cfg.nowMs - nextStart) / cfg.durationMs);

  if (nextLeftNow < prevRightNow - tolerance) return true;

  const prevRightEnd = prevRightNow - prevDistance;
  const nextLeftEnd = nextLeftNow - nextDistance;
  return nextLeftEnd < prevRightEnd - tolerance;
}

9.2 顶部/底部固定轨道

固定弹幕不做横向碰撞，只按行停留时间复用：

function findFixedRow(item, rows, nowMs, fromBottom, config) {
  for (const row of rows) {
    if ((!row.placement || row.placement.expireMs <= nowMs) && item.height <= row.height) {
      return row;
    }
  }

  return createFixedRow(item.height, rows, fromBottom, config);
}

顶部从上往下排，底部从下往上排。固定弹幕同屏数量在高密度视频里很敏感，建议对顶部/底部单独设预算。

10. 字体、颜色、描边

官方默认字体族是：

Linux: 'Noto Sans CJK SC DemiLight'
Others: SimHei, 'Microsoft JhengHei'

Native 复刻建议：

class DanmakuTextStyle(
  val fontFamily: String = "sans-serif-medium",
  val bold: Boolean = true,
  val opacity: Float = 0.8f,
  val fontScale: Float = 1.0f,
  val borderMode: Int = 0
)

fun resolveTextSize(rawSize: Int, density: Float, fontScale: Float): Float {
  val base = rawSize.coerceIn(12, 25)
  return base * density * fontScale
}

描边策略建议按文字亮度反色。MyBLBL 当前已经有这类逻辑，可保留并调参：

fun resolveStrokeColor(textColor: Int): Int {
  val r = red(textColor)
  val g = green(textColor)
  val b = blue(textColor)
  val luminance = (0.2126f * r + 0.7152f * g + 0.0722f * b) / 255f
  return if (luminance > 0.5f) {
    argb(230, 0, 0, 0)
  } else {
    argb(210, 255, 255, 255)
  }
}

fun resolveStrokeWidth(textSizePx: Float): Float {
  return (textSizePx * 0.09f).coerceIn(1.5f, 3.0f)
}

颜色处理要注意 B 站 protobuf 里通常是 RGB，没有 alpha。内部颜色应补成 ARGB：

fun resolveTextColor(rgb: Int, allowColor: Boolean): Int {
  val argb = if (rgb == 0) Color.WHITE else (rgb or 0xFF000000.toInt())
  return if (allowColor) argb else Color.WHITE
}

11. 渲染架构选择

B 站 Web 端弹幕层是独立覆盖层，不参与视频解码。复刻到 Android/TV 时建议维持 MyBLBL 现在的结构：视频播放器一层，弹幕 View 一层。

方案优点风险
DOM/CSSWeb 端简单，CSS 动画方便。Android native 不适合。
Canvas 逐帧文字实现直接。高密度时 CPU 压力大。
Bitmap 缓存 + Canvas 绘制适合 TV/Android，缓存命中后很快。缓存 miss、内存和 GC 要严格控制。
OpenGL/Texture极限性能更好。复杂度高，文字纹理管理难。

当前 MyBLBL 的方向是正确的：测量、构建 DrawingCache、帧命令、缓存命中绘制、fallback 直绘。要复刻 B 站体验，重点不是推倒重来，而是把数据减量、首屏分段、过滤和密度控制做得更像 B 站。

12. 缓存与性能预算

建议把弹幕运行时拆成两条线程/队列：

  • 主/渲染线程：只做轻量调度、轨道坐标、draw bitmap。
  • 缓存线程：测量文本、生成 bitmap、处理 VIP 纹理。
class RenderRuntime {
  fun tick(nowMs: Long) {
    syncNewData()
    expireOldItems(nowMs)
    enqueueDueItems(nowMs, budget = currentBudget())
    scheduleMeasureAndCache(nowMs, maxMeasure = 12, maxCache = 8)
    buildFrameCommands(nowMs)
  }

  fun draw(canvas: Canvas) {
    for (cmd in currentFrame.commands) {
      if (cmd.cache != null) {
        canvas.drawBitmap(cmd.cache, cmd.x, cmd.y, alphaPaint)
      } else if (fallbackBudget.tryAcquire(cmd.mode)) {
        renderer.drawTextDirect(canvas, cmd.item)
      }
    }
  }
}

建议预算：

项目建议值说明
每帧入队正常 32，过载 24/12/6负载保护只降速，不永久丢弹幕。
每帧测量12和 MyBLBL 当前一致，可根据设备动态调。
每帧缓存请求8避免缓存线程堆积。
fallback 直绘滚动 12，固定 8超出后跳过，优先保帧率。
缓存池1080p 50MB，4K 72MB 起TV 端要按内存动态调整。

13. 重复弹幕合并

这个样本视频重复内容很多，例如抽奖相关短词。重复合并可以显著降低视觉密度和轨道压力。

data class MergeKey(
  val text: String,
  val mode: Mode,
  val color: Int,
  val colorfulType: Int
)

fun mergeDuplicates(items: List<DanmakuModel>, windowMs: Long = 2000): List<DanmakuModel> {
  val result = mutableListOf<DanmakuModel>()
  val bucket = LinkedHashMap<MergeKey, MutableList<DanmakuModel>>()

  for (item in items.sortedBy { it.timeMs }) {
    flushExpired(bucket, item.timeMs - windowMs, result)
    val key = MergeKey(item.text.trim().lowercase(), item.mode, item.colorRgb, item.colorfulType)
    bucket.getOrPut(key) { mutableListOf() }.add(item)
  }

  flushAll(bucket, result)
  return result.sortedBy { it.timeMs }
}

展示方式建议：

  • 2 条重复：显示原文或轻量后缀。
  • 3 条以上：显示 文本 xN。
  • 合并后的权重取最大值，时间取第一条或中位数。
  • 合并后要重新测量宽度，避免碰撞计算错误。

14. 复刻架构蓝图

推荐不要把 B 站 Web 的 DOM 引擎直接搬到 Android。正确做法是复刻它的流水线和策略：服务端分段、默认过滤、密度预算、轨道排布、独立覆盖层渲染。MyBLBL 现有 native 弹幕框架已经具备基础，重点是补上“进入渲染前的减量”和“高峰期预算”。

15. API 与加载状态机

复刻 B 站体验时，首屏加载顺序比渲染器更重要。卡顿高发点通常是：一次性解析 360 秒高密度分段、一次性 submit 大窗口、缓存 miss 后每帧直绘过多。

enum class LoadState {
  Idle,
  LoadingView,
  LoadingFirstSlice,
  ReadyFirstScreen,
  Prefetching,
  Seeking,
  Error
}

class BiliDanmakuLoader(
  private val api: DanmakuApi,
  private val parser: DmProtoParserFacade
) {
  private val loaded = LinkedHashMap<SegmentKey, List<DanmakuModel>>()
  private var view: DmViewInfo? = null

  suspend fun open(aid: Long, cid: Long, durationSec: Int): List<DanmakuModel> {
    view = parser.parseView(api.dmView(aid, cid, durationSec))

    // 关键：首屏只拿 0~120s，不先吃完整 360s。
    val first = loadSlice(aid, cid, segmentIndex = 1, ps = 0, pe = 120_000)
    loaded[SegmentKey.Slice(1, 0, 120_000)] = first
    return first
  }

  suspend fun ensureAround(aid: Long, cid: Long, positionMs: Long) {
    when {
      positionMs < 90_000L -> {
        // 90s 前预取 120~360s，避免 120s 附近断层。
        prefetchSlice(aid, cid, 1, 120_000, 360_000)
      }
      else -> {
        val index = (positionMs / 360_000L).toInt() + 1
        prefetchSegment(aid, cid, index)
        prefetchSegment(aid, cid, index + 1)
      }
    }
  }
}

实现细节：

  • SegmentKey 要区分完整 segment 和首段 slice，否则 segment_index=1 会误判已加载。
  • seek 到未加载区间时，优先加载当前位置所在 segment，再加载下一个 segment。
  • 解析结果进入 timeline 前先过滤，不要让低质量/被屏蔽弹幕进入测量队列。
  • append 新数据时要保持全局按 timeMs 排序，可用二分插入或归并，不要每次全量 sort 大列表。

16. 过滤链代码骨架

过滤链要放在 protobuf 映射之后、提交给 DanmakuPlayer 之前。目标是让渲染层只处理“可能真的会显示”的弹幕。

data class BiliFilterSetting(
  val aiSwitch: Boolean = true,
  val aiLevel: Int = 3,
  val showScroll: Boolean = true,
  val showTopBottom: Boolean = true,
  val showColor: Boolean = true,
  val showSpecial: Boolean = true,
  val preventShade: Boolean = false,
  val seniorMode: Boolean = false,
  val reportRegex: List<Regex> = emptyList()
)

fun shouldDropByBiliRule(dm: DanmakuModel, setting: BiliFilterSetting): Boolean {
  if (dm.text.isBlank()) return true
  if (setting.reportRegex.any { it.containsMatchIn(dm.text) }) return true

  if (setting.aiSwitch && kotlin.math.abs(dm.weight) < setting.aiLevel) {
    return true
  }

  if (setting.seniorMode && dm.weight <= 10) return true
  if (!setting.showScroll && dm.mode == Mode.Rolling) return true
  if (!setting.showTopBottom && (dm.mode == Mode.Top || dm.mode == Mode.Bottom)) return true
  if (!setting.showColor && dm.isColorDanmaku()) return true
  if (!setting.showSpecial && dm.isSpecialDanmaku()) return true
  if (setting.preventShade && dm.mode == Mode.Bottom) return true

  return false
}

fun DanmakuModel.isColorDanmaku(): Boolean {
  if (colorfulType != 0) return true
  val rgb = colorRgb and 0x00ffffff
  return rgb != 0x00ffffff
}

fun DanmakuModel.isSpecialDanmaku(): Boolean {
  if (pool == 2) return true
  if (!animation.isNullOrBlank()) return true
  return mode == Mode.Advanced || mode == Mode.Script || mode.raw >= 1000
}

和当前 MyBLBL 的区别：当前 applySmartFilter 使用“本地等级 + 最大正分”推导阈值，这适合本地调节，但和 B 站 Web 的 abs(weight) < aiLevel 不同。建议提供一个“B 站兼容模式”，默认用官方逻辑，再允许用户手动调强。

17. 轨道与速度的复刻细节

滚动弹幕的体验由三个值决定：运动距离、持续时间、同轨避碰。B 站 Web 的视觉效果不是“固定像素速度”，而是每条弹幕从右侧外部移动到左侧外部，文本越长总移动距离越长。

fun rollingLeft(
  screenWidth: Float,
  itemWidth: Float,
  nowMs: Long,
  startMs: Long,
  durationMs: Long
): Float {
  val elapsed = (nowMs - startMs).coerceAtLeast(0L).toFloat()
  val progress = (elapsed / durationMs).coerceIn(0f, 1f)
  return screenWidth - progress * (screenWidth + itemWidth)
}

fun resolveRollingDuration(baseMs: Long, speedplus: Float, playbackSpeed: Float): Long {
  // B 站设置里 speedplus 越大视觉上越快，duration 应越短。
  val speed = (speedplus * playbackSpeed).coerceIn(0.5f, 3.0f)
  return (baseMs / speed).toLong().coerceIn(3_500L, 12_000L)
}

同轨避碰要满足两个条件：

  • 当前时刻：后一条进入时不能贴上前一条尾部。
  • 结束时刻：后一条速度更快或更短时，不能在离屏前追上前一条。
fun canPlaceAfterTail(
  tail: RollingPlacement,
  nextWidth: Float,
  nextStartMs: Long,
  nowMs: Long,
  screenWidth: Float,
  durationMs: Long,
  gapPx: Float
): Boolean {
  val tailLeftNow = rollingLeft(screenWidth, tail.motionWidth, nowMs, tail.startMs, durationMs)
  val tailRightNow = tailLeftNow + tail.width
  val nextLeftNow = rollingLeft(screenWidth, nextWidth, nowMs, nextStartMs, durationMs)
  if (nextLeftNow < tailRightNow + gapPx) return false

  val checkAt = minOf(tail.startMs + durationMs, nextStartMs + durationMs)
  val tailRightEnd = rollingLeft(screenWidth, tail.motionWidth, checkAt, tail.startMs, durationMs) + tail.width
  val nextLeftEnd = rollingLeft(screenWidth, nextWidth, checkAt, nextStartMs, durationMs)
  return nextLeftEnd >= tailRightEnd + gapPx
}

MyBLBL 当前 RollingCollision 已经接近这个方向，后续主要补两点：把 speedplus 接进 rollingDurationMs，并且在高密度时按滚动/固定分别收紧入队预算。

18. 字号、颜色、描边实现细节

B 站 protobuf 的 size 常见值是 25 和 18。Web 默认 fontsize=1，播放器设置会整体缩放。Native 侧建议保持“协议字号”和“用户缩放”分离。

fun resolveDanmakuTextSizePx(
  protocolSize: Int,
  density: Float,
  userFontScale: Float
): Float {
  val normalizedSp = when {
    protocolSize <= 18 -> 18f
    protocolSize >= 25 -> 25f
    else -> protocolSize.toFloat()
  }
  return normalizedSp * density * userFontScale
}

fun resolveArgbColor(protocolRgb: Int, opacity: Float): Int {
  val rgb = protocolRgb.takeIf { it != 0 } ?: 0x00ffffff
  val alpha = (255 * opacity.coerceIn(0f, 1f)).toInt()
  return (alpha shl 24) or (rgb and 0x00ffffff)
}

fontborder 建议映射成四种本地模式。B 站 Web 的具体绘制由 CSS/Canvas/文本节点组合承载，Native 不需要照搬，只要复刻视觉结果：

fontborder建议效果Native 绘制
0默认描边先 stroke 后 fill，描边颜色按亮度反色。
1重描边strokeWidth 提升到 textSize * 0.12。
2投影setShadowLayer 或离屏 bitmap 阴影。
3无描边只 fill，建议仅在低密度或用户指定时使用。
 
fun drawTextWithBorder(
  canvas: Canvas,
  text: String,
  x: Float,
  baseline: Float,
  fillPaint: TextPaint,
  strokePaint: TextPaint,
  borderMode: Int
) {
  when (borderMode) {
    3 -> canvas.drawText(text, x, baseline, fillPaint)
    2 -> {
      fillPaint.setShadowLayer(2.5f, 1f, 1f, Color.BLACK)
      canvas.drawText(text, x, baseline, fillPaint)
      fillPaint.clearShadowLayer()
    }
    else -> {
      strokePaint.strokeWidth = if (borderMode == 1) {
        (fillPaint.textSize * 0.12f).coerceIn(2f, 4f)
      } else {
        (fillPaint.textSize * 0.09f).coerceIn(1.5f, 3f)
      }
      canvas.drawText(text, x, baseline, strokePaint)
      canvas.drawText(text, x, baseline, fillPaint)
    }
  }
}

MyBLBL 当前 SimpleRenderer 已经实现了标准描边、亮度反色、VIP 渐变和测量缓存。建议只增加 fontborder 入参和透明度统一处理，不要重写渲染器。

19. 防挡弹幕的接入方式

智能防挡不应该参与普通轨道分配。它是渲染裁剪层：弹幕仍按原始轨道和坐标计算，只是在绘制时根据 mask 决定哪些像素或区域不显示。

视频帧/模型输出 mask
  - mask 坐标归一化到视频内容区域
  - 处理播放器黑边、缩放、旋转
  - 输出到 DanmakuOverlay 的 clip/mask 数据

弹幕运行时
  - 不改变弹幕 time、track、x/y
  - 不因为 mask 重新排轨
  - 绘制阶段应用 mask

Canvas 复刻策略
  - 简化版：按矩形/路径 region 裁剪
  - 进阶版：mask bitmap + DST_OUT / shader
  - 高性能版：OpenGL texture mask

这样做的原因是：防挡是视频内容感知，轨道分配是时间轴排布。把两者耦合会导致 seek、resize、mask 延迟时弹幕位置抖动。当前 MyBLBL 的 DmMaskController 应继续作为独立控制器，只给渲染层提供 mask。

21. 推荐落地顺序

  1. 补齐官方 view/segment 字段解析，尤其是 dmSetting、reportFilter、dmRestrict。
  2. 实现官方首屏分段请求：0~120s、120~360s、后续标准 segment。
  3. 把默认设置改成 B 站 Web 近似值：区域 50%、透明度 0.8、AI 3、密度 normal。
  4. 增加官方过滤兼容模式：abs(weight) < aiLevel、类型过滤、颜色过滤、特殊弹幕过滤。
  5. 保留现有窗口化和 bitmap 缓存，但把入队预算、测量预算、fallback 指标暴露出来。
  6. 对固定弹幕增加单独预算，因为样本视频顶部固定弹幕占比高。
  7. 最后再做高级弹幕、彩色纹理和智能防挡联动。

23. 验收指标

复刻是否接近 B 站，不要只靠肉眼。建议每个视频采样以下指标：

指标目标异常说明
首屏弹幕可见时间< 500ms过慢说明首段请求/解析/测量太重。
单帧 update< 8ms，过载 < 12ms超过会影响视频合成。
缓存命中率> 90%低于目标会触发直绘和跳过。
fallback skipped持续为 0 或低频尖峰持续增长说明缓存跟不上。
active item 数稳定在设备可承受范围尖峰说明窗口/密度策略失效。
固定弹幕拒绝数高峰可接受，不能持续满拒绝顶部/底部预算需要调。

24. 关键结论

数据层B 站不卡，首先是因为它不会默认把历史 10 万条全给客户端。view/segment 返回的是被服务端和设置共同约束后的展示集合。

调度层首屏拆分 0~120s 是关键优化。复刻时应优先做这个，而不是先改渲染器。

过滤层abs(weight) < aiLevel 是官方 AI 过滤的核心可见逻辑之一，应作为兼容模式加入。

渲染层MyBLBL 当前 bitmap 缓存方向正确。真正需要补的是默认密度、过滤、首段调度和固定弹幕预算。
